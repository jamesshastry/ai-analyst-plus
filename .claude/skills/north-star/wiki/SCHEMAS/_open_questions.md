The 4 machine indices (`wiki/QUERY_INDEX.yaml`, `wiki/CASES_INDEX.yaml`, `wiki/GLOSSARY.yaml`, `wiki/QUOTES.yaml`) will be schema-validated by `index-builder`'s internal checks rather than via standalone schema YAMLs in this directory.

## Q2. Schema YAML format: meta-schema vs JSON Schema Draft 2020-12

**Source of conflict:** The user prompt suggests JSON Schema Draft 2020-12 expressed in YAML (with `$schema:`, `$id:`, `properties:`, `additionalProperties: true`). The agent prompt specifies a custom meta-schema (with `schema_name`, `required_fields`, `type_specific_required`, `enums`, `id_pattern`, `extends`).

**Conservative interpretation taken:** Followed the agent prompt's meta-schema format. Rationale (also captured in `_README.md`):
- `_lib/validators.py` is the only consumer — no jsonschema runtime dependency needed.
- The meta-schema maps 1:1 to the validation rules described in `WIKI_SCHEMA.md`, making cross-audit easier.
- `_lib/schema_export.py` (if added later) can mechanically transform to JSON Schema for any consumer that needs it.

**Action if needed:** If a downstream consumer requires JSON Schema, write a one-time mechanical transformer rather than rewriting all 28 schemas by hand.

## Q3. `ProTipAtom.concept_id` cross-link before concepts exist

**Spec gap:** `WIKI_SCHEMA.md` requires PRO TIP atoms to reference a `concept_id`, but the concept-extractor produces concepts AND pro-tips in the same agent run. Ordering within a single agent run isn't specified.

**Conservative interpretation taken:** Required `concept_id` on every ProTipAtom; flagged `cross_check: existing_atom_id` in `field_constraints`. If concept-extractor emits pro-tips before the concepts they reference, validators.py must validate `concept_id` only after the full atom set is written (not per-file). This is the existing pattern for `related_atoms` and is assumed by `validators.py`.

**Action if needed:** Document the two-pass validation order in `validators.py` (Pass 1: per-file structural; Pass 2: cross-atom references after all atoms emitted).

## Q4. `CaseStudyFragmentAtom.fragment_role` not in source spec

**Conservative interpretation taken:** Added `fragment_role` enum [`nsm`, `inputs`, `evolution`, `outcome`, `context`, `counterfactual`] as `type_specific_required` on `CaseStudyFragmentAtom.yaml`. This lets `cases-assembler` deterministically pick which fragment populates each slot in the case-article frontmatter without re-parsing prose.

**Action if needed:** If example-harvester's emit pattern differs, either expand the enum or relax to `type_specific_optional` and let cases-assembler fall back to LLM-classification.
